### Core Topics and Key Insights

#### 1. **Incident Response Lifecycle and Frameworks**

- The course focuses on the **NIST Cybersecurity Framework (CSF)** phases: Detect, Respond, and Recover.
- The **NIST Incident Response Lifecycle** includes five phases:
    - Preparation
    - Detection and Analysis
    - Containment, Eradication, and Recovery
    - Post-Incident Activity
- The lifecycle is **cyclical and non-linear**, meaning phases may overlap as investigations progress.
- An **incident** is defined by NIST as any event that jeopardizes confidentiality, integrity, or availability of information or violates security policies.

#### 2. **Incident vs. Event**

- **Events** are observable occurrences (e.g., user password reset).
- **Incidents** are events involving unauthorized or malicious activity (e.g., attacker changing a password unlawfully).
- All incidents are events, but not all events are incidents.

#### 3. **Incident Handler’s Journal**

- Analysts document the **Five Ws** of incidents: Who, What, When, Where, and Why.
- Documentation ensures transparency, accountability, and assists in investigations and reporting.

#### 4. **Incident Response Teams (CSIRTs)**

- CSIRTs are cross-functional teams responsible for managing security incidents.
- Key roles include:
    - **Security Analyst**: Investigates alerts and determines incident severity.
    - **Technical Lead**: Provides technical guidance on incident lifecycle.
    - **Incident Coordinator**: Manages team activities and communication.
- Team structures and names vary by organization but share the common goal of incident management.

#### 5. **Incident Response Plans**

- Formal plans outline **policies, standards, and procedures** to ensure consistent, efficient incident handling.
- Plans include system/network information, contact lists, and templates.
- Regular testing via tabletop exercises or simulations is crucial for preparedness and regulatory compliance.

### Detection and Monitoring Tools

| Tool/Technology                                       | Purpose/Function                                                                                   | Key Examples                 |
| ----------------------------------------------------- | -------------------------------------------------------------------------------------------------- | ---------------------------- |
| Intrusion Detection System (IDS)                      | Monitors network/system activity and alerts on suspicious behavior (detection only).               | Snort, Zeek, Sagan, Suricata |
| Intrusion Prevention System (IPS)                     | Monitors and takes action to block malicious activity (detection + prevention).                    | Suricata                     |
| Security Information and Event Management (SIEM)      | Aggregates, normalizes, analyzes logs and alerts from multiple sources for centralized monitoring. | Splunk, Google Chronicle     |
| Security Orchestration Automation and Response (SOAR) | Automates response workflows and case management.                                                  | _Not specified_              |
| Packet Sniffers/Analyzers                             | Capture and analyze network packets to identify potential indicators of compromise (IOC).          | tcpdump, Wireshark           |

- **SIEMs** function like a car dashboard, providing real-time visibility and alerting on network security events by analyzing large volumes of heterogeneous log data.
- **SOAR** tools automate repetitive tasks and help manage multiple incidents as cases.

### Network Traffic and Packet Analysis

- **Network traffic** represents data flows across networks; abnormal patterns can indicate incidents.
- **Indicators of Compromise (IOC)**, such as unusual outbound data volumes, aid in spotting data exfiltration and other attacks.
- Network packets consist of:
    - **Header**: Protocol info, IP addresses, ports (like an envelope’s address).
    - **Payload**: Actual data content.
    - **Footer**: Packet termination.
- Packet sniffers capture packets, creating **Packet Capture (Pcap)** files for analysis.
- Analysts manually dissect packets (especially IP headers like IPv4) to understand traffic and identify anomalies.
- Command-line tool **tcpdump** and GUI tool **Wireshark** are commonly used for packet capture and analysis.

### Incident Detection and Analysis

- Detection involves verifying alerts generated by systems like IDS, IPS, and SIEM.
- Analysts face challenges such as alert fatigue and false positives resulting from broad or misconfigured rules.
- Effective alert triage prioritizes incidents based on urgency, following organizational policies.
- Analysts gather contextual information (time, source IP, frequency) to validate alerts.

---

### Incident Containment, Eradication, and Recovery

- **Containment** limits damage (e.g., isolating infected systems).
- **Eradication** removes threats (e.g., patching vulnerabilities, deleting malware).
- **Recovery** restores normal operations (e.g., system reimaging, password resets).
- These phases are interrelated and iterative.

---

### Post-Incident Activities

- Involves **final reporting** and **lessons learned meetings** to review incidents and improve response.
- Documentation is critical for transparency, compliance, and legal proceedings.
- Emphasis on non-blaming culture to foster learning from human errors and process gaps.

### Documentation in Incident Response

|Documentation Type|Purpose|
|---|---|
|Incident Handler’s Journal|Notes during investigations capturing the Five Ws to track incident details.|
|Playbooks|Step-by-step manuals for responding to specific incident types to reduce confusion and speed response.|
|Chain of Custody Forms|Track evidence handling to ensure integrity, especially for legal compliance.|
|Final Incident Reports|Comprehensive incident timeline, actions taken, and recommendations for prevention.|

- Documentation improves **transparency**, **standardization**, and **clarity** during incident response.
- Tools for documentation include Google Docs, OneNote, Jira (ticketing), audio recorders, and handwritten notes.

### Log Analysis and Signatures

- Logs are records of events generated by systems, devices, and applications, critical for incident investigations.
- Common log formats include:
    - **Syslog** (protocol and format)
    - **JSON** (key-value pairs, human-readable)
    - **XML** (tagged data format)
    - **CSV** (comma-separated values)
- Logs may be verbose; selective logging optimizes investigation efficiency.
- **Signature-based detection** uses predefined rules to identify suspicious activity (e.g., multiple failed logins).
- Security analysts can write, customize, and test IDS signatures to reduce false positives and improve detection.
- **Suricata**, a signature-based IDS, outputs alerts and network telemetry in **EVE JSON** format.

---

### Searching and Querying SIEM Tools

- SIEM platforms like **Splunk** and **Google Chronicle** provide search capabilities to locate and analyze events.
- Effective queries:
    - Use specific keywords, event IDs, and time ranges to narrow results.
    - Utilize Boolean operators and wildcards (e.g., OR, * wildcard in Splunk’s SPL).
    - Use UDM (Unified Data Model) searches in Chronicle for normalized data.
- Searching raw logs may be necessary to find unnormalized or troubleshooting data.
